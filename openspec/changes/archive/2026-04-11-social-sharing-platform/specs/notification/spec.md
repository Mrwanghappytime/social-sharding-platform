## 新增需求

### 需求:通知持久化
系统必须将所有通知持久化到数据库，包括评论、点赞、关注三种类型。

#### 场景:评论通知持久化
- **当** 用户收到评论时
- **那么** 系统将通知写入数据库

#### 场景:点赞通知持久化
- **当** 用户收到点赞时
- **那么** 系统将通知写入数据库

#### 场景:关注通知持久化
- **当** 用户收到关注时
- **那么** 系统将通知写入数据库

### 需求:普通用户实时推送
系统必须通过 WebSocket + Redis Pub/Sub 实时推送通知给在线的普通用户（粉丝数 < 10000）。

#### 场景:在线普通用户收到点赞通知
- **当** 某用户（粉丝数 < 10000）收到点赞通知
- **那么** 系统通过 WebSocket 实时推送通知到用户端

#### 场景:在线普通用户收到评论通知
- **当** 某用户（粉丝数 < 10000）收到评论通知
- **那么** 系统通过 WebSocket 实时推送通知到用户端

#### 场景:在线普通用户收到关注通知
- **当** 某用户（粉丝数 < 10000）收到关注通知
- **那么** 系统通过 WebSocket 实时推送通知到用户端

### 需求:KOL 用户延迟推送
系统必须对 KOL 用户（粉丝数 >= 10000）的通知不进行实时推送，用户上线时从数据库拉取未读通知。

#### 场景:KOL 用户上线拉取未读通知
- **当** KOL 用户上线并请求未读通知
- **那么** 系统从数据库返回所有未读通知

#### 场景:KOL 用户标记通知已读
- **当** KOL 用户查看通知后
- **那么** 系统将通知标记为已读

### 需求:获取通知列表
系统必须支持获取当前用户的通知列表，按创建时间倒序排列。

#### 场景:获取通知列表
- **当** 请求通知列表
- **那么** 系统返回当前用户的通知列表

#### 场景:分页获取通知
- **当** 请求带分页参数的通知列表
- **那么** 系统返回对应分页的通知列表

### 需求:标记通知已读
系统必须支持将单条或全部通知标记为已读。

#### 场景:标记单条已读
- **当** 用户查看某条通知
- **那么** 系统将该通知标记为已读

#### 场景:标记全部已读
- **当** 用户请求标记全部通知已读
- **那么** 系统将用户的所有未读通知标记为已读

### 需求:获取未读通知数
系统必须支持获取当前用户的未读通知数量。

#### 场景:获取未读数
- **当** 请求未读通知数量
- **那么** 系统返回未读通知数
